# S1：单球直线进袋 RL 环境搭建与训练说明

## 文档简介
这份文档只服务于 **S1 最小闭环**。目标很单纯：先把 `单球直线进袋 -> 单步环境 -> baseline -> PPO -> 评估与日志` 这一整条链路跑通。

文档分成五个板块：

1. **阶段目标与验收标准**：明确 S1 做到什么才算完成。
2. **环境设计**：定义球型、状态、动作、奖励、终止条件、事件解析。
3. **训练与 baseline**：先验证环境可解，再接 PPO。
4. **观测与记录**：规定 success rate、eval、可视化、日志、checkpoint 的保存方式。
5. **交付物与执行顺序**：确保 S1 结束后能沉淀成可复用资产。

这版额外把初始球位固定成两个标准 preset：

- `eight_ball_reference`：参考 8-ball 的标准球桌几何基准位
- `five_point_straight_in`：参考常见“五分点”直线练习位

这样做的好处是：S1 的球型不再是随意手摆，而是从一开始就绑定到常见训练位和规则参考位，更容易 debug，也更便于后续扩展。8-ball 的标准开局里，15 颗目标球会被紧密摆成三角形，最前端球对准脚点，8 号球位于球堆中心，母球开球时在头线后方的开球区内。常见“五分点”直线练习则通常把目标球放在中线附近固定点，母球放在开球线附近，并让母球、目标球和袋口近似三点一线。

---

## 1. 阶段目标

S1 的目标不是做完整桌球智能体，而是把 **最小可训练 RL 闭环** 跑通：

- 场景只保留：`母球 + 1 个目标球 + 1 个目标袋口`
- 每个 episode 只打一杆
- agent 只输出最小动作：`phi`（出杆方向）和 `V0`（出杆力度）
- 环境调用一次 `simulate`
- 根据是否进袋、是否犯规、是否失败返回 reward

这是整个 PPO 训练路线里最关键的第一步。S1 做对了，后面 S2 的随机角度进袋、S3 的停位控制、S4 的 PPO 基线都只是扩展，而不是推倒重来。

---

## 2. S1 的成功标准

建议把 S1 的验收标准定成下面这几条：

1. 能稳定 reset 出固定单球局面
2. 能把 agent 输出的 `phi, V0` 正确映射到 pooltool 的 cue 参数
3. 能在一次 `step()` 后完成整杆模拟并正确判定终局
4. 有至少 2 个 baseline：
   - random baseline
   - heuristic / search baseline
5. 在固定球型上，baseline 明显优于随机
6. PPO 训练后，固定球型上的 `success rate >= 80%`
7. 训练日志、评估结果、checkpoint、配置文件都能被完整保存

## 3. S1 需要做的事情

### 3.1 固定两套起始球位 preset

S1 不建议“随机摆一个直线球”，而是直接固定成两个 preset。

#### Preset A：`eight_ball_reference`

用途：

- 给 S1 一个严格规则导向的标准参考球位
- 让后续 S2/S3/S4 更容易和真实球桌几何接轨

实现建议：

- 桌面坐标系使用 pooltool 的标准球桌坐标
- 目标袋口固定为某个底袋，例如右下角底袋
- 目标球放在 **脚点附近** 的固定位置
- 母球放在 **头线后方** 的固定出杆位
- 母球、目标球、目标袋口三点尽量构成接近直线的简化练习位

解释：

- 真实 8-ball 开局中，球堆最前端球对准脚点，8 号球在球堆中心，母球在头线后开球区内。
- 但 S1 只保留一个目标球，不保留整堆 15 球。所以这里不复现完整球堆，而是**借用 8-ball 的基准点**来定义单球参考局面。

最实用的落地方式是：

- 将 `object_ball_pos` 设为 `foot_spot` 或其附近一个可直线进袋的位置
- 将 `cue_ball_pos` 设为 `behind_head_string` 的固定点
- 将 `target_pocket` 固定为一个底袋

这样可以保留 8-ball 的桌面几何语义，但又不会把 S1 复杂化。

#### Preset B：`five_point_straight_in`

用途：

- 作为更典型的基本功直线球训练位
- 让 PPO 先在最容易学的几何关系上起步

实现建议：

- 目标球放在球桌中线附近的固定点
- 母球放在头线或开球线附近固定点
- 目标袋口固定为与目标球成直线的底袋或中袋
- 明确要求：母球、目标球、袋口三点近似共线

解释：

- 常见“五分点”训练描述本质上就是固定目标球位和固定母球位的直线练习，目的是练准度与出杆稳定性。
- 对 S1 来说，这种布局比“完全随机直线球”更可控，也更适合做 curriculum 的起点。

#### 推荐顺序

建议训练顺序如下：

1. 先只跑 `five_point_straight_in`
2. PPO 在该 preset 上稳定后，再加入 `eight_ball_reference`
3. 最后再让两种 preset 混合采样

这样最稳。

### 3.2 环境定义

需要先把最小环境定义文档写清楚。

#### Observation

S1 建议只用低维状态，不要上图像。

可先用：

- 母球位置 `(x_cue, y_cue)`
- 目标球位置 `(x_obj, y_obj)`
- 目标袋口位置 `(x_pocket, y_pocket)`
- 可选：母球到目标球向量
- 可选：目标球到袋口向量
- 可选：当前 preset id 的 one-hot 或整数编码

推荐先做成一个固定长度的数值向量。这样最稳。

#### Action

S1 先只保留两个连续动作：

- `phi`：击球方向角
- `V0`：出杆初速度 / 力度

不要一开始就放：

- `a, b`（左右/上下塞）
- `theta`（杆抬角）

动作空间越小，PPO 越容易学。

#### Reward

S1 推荐从极简 reward 开始：

- 目标球进目标袋：`+1.0`
- 母球落袋：`-1.0`
- 目标球没进袋：`0.0` 或 `-0.01`
- 非法情况 / 模拟失败：`-1.0`

一开始不要把 reward 设计得太花。

S1 的核心目标是：**先让 agent 学会“进球”**，而不是学复杂技巧。

#### Termination

S1 建议一杆即终局：

- 成功进袋 -> done
- 母球落袋 / 犯规 -> done
- 没进袋 -> done
- 模拟异常 -> done

这会把问题变成一个很干净的单步 RL / contextual bandit 式任务。训练最稳。

### 3.3 几何约束与局面生成

S1 虽然叫“单球直线进袋”，但也不能随便摆球。需要控制好局面生成。

#### 最推荐的起步方式

先只用固定球型：

- 只启用 `five_point_straight_in`
- 目标球、母球、袋口严格接近直线
- 中间无遮挡
- 不引入库边反弹、塞球、跳球等复杂因素

#### 后续再加的小随机化

当固定球型能稳定成功后，再逐步加入：

- 母球位置的小幅扰动
- 目标球位置的小幅扰动
- 力度容忍范围测试
- 袋口切换
- `five_point_straight_in` 与 `eight_ball_reference` 的混合采样

注意：S1 不是 S2。S1 的首要任务是把闭环跑通，不是做泛化。

---

### 3.4 事件解析与成败判定

你必须先把 pooltool 的事件解析逻辑单独验证清楚。

S1 至少要能回答下面几个问题：

1. 目标球是否进了指定袋口
2. 母球是否落袋
3. 第一碰是否正确命中目标球
4. 最终两颗球是否都停下
5. 本次模拟是否正常结束

建议先手写脚本，不接 RL，人工指定几组 `phi, V0`，然后打印：

- event 列表
- 每颗球最终位置
- 是否进袋
- 是否犯规

只有这些都看懂了，再封装成 `env.step()`。

---

### 3.5 baseline 先行

S1 阶段不要一上来就 PPO。

先做两个 baseline。

#### Baseline A：random

随机采样 `phi, V0`，在固定球型上跑若干次，得到：

- success rate
- foul rate
- reward 均值

这提供最低基线。

#### Baseline B：heuristic / search

至少做一个简单启发式：

- `phi` 沿着“母球 -> 目标球 -> 袋口”几何关系近似计算
- `V0` 取一个经验范围
- 或者在 `phi, V0` 小范围网格搜索

这个 baseline 的作用非常关键：

- 证明环境是可解的
- 证明 reward 和事件解析没写坏
- 给 PPO 设立可比目标

如果 heuristic 都打不进，先不要训 PPO，先查环境。

## 4. 如何搭建 S1 单球直线进袋 RL 环境

下面是推荐搭建顺序。

### 4.1 先做一个不带 Gym 的裸环境原型

先不要急着接 Gymnasium / Stable-Baselines3。

先写一个最小原型类，至少包含：

- `build_layout(preset_name)`：创建桌面、球、袋口、母球、目标球
- `reset()`：按 preset 重置球型并返回 obs
- `step(action)`：
  - 把 `phi, V0` 映射到 cue 参数
  - 调用一次 `simulate`
  - 解析事件
  - 计算 reward / done / info
- `render_debug()`：可视化或导出一杆结果

这样更容易查错。

### 4.2 再包装成标准 Gymnasium 环境

当裸环境逻辑正确后，再包成标准接口：

- `observation_space = Box(...)`
- `action_space = Box(...)`
- `reset(seed=...) -> obs, info`
- `step(action) -> obs, reward, terminated, truncated, info`

S1 因为一杆即终局，所以 `step()` 之后通常直接结束。

### 4.3 action 映射必须显式做缩放

不要让 policy 直接输出物理量原值。

建议让 policy 输出归一化动作，再映射成物理参数：

- `phi_norm in [-1, 1]` -> `phi in [phi_min, phi_max]`
- `v_norm in [-1, 1]` -> `V0 in [v_min, v_max]`

这样能减少训练不稳定。

并且建议：

- `phi` 不要一开始就允许全 360°
- 对于 `five_point_straight_in`，先只给较窄角度窗口
- `V0` 也限制在合理物理范围内，避免过轻或过猛

### 4.4 info 字典要设计好

每次 `step()` 除了 reward，还要返回详细 `info`，至少包含：

- `preset_name`
- `layout_id`
- `success`: 是否目标球进目标袋
- `cue_scratch`: 母球是否落袋
- `legal_first_hit`: 第一碰是否合法
- `phi`: 实际执行的角度
- `V0`: 实际执行的力度
- `obj_final_pos`: 目标球最终位置
- `cue_final_pos`: 母球最终位置
- `events_summary`: 关键事件摘要

这对 debug 和日志记录很重要。

## 5. 训练时需要注意什么

### 5.1 先固定环境，再训练

训练前先确认以下四件事：

1. 固定 seed 下，reset 出来的局面一致
2. 同一动作输入下，模拟结果基本一致
3. heuristic baseline 明显优于 random
4. reward 与 success 判定一致，不会出现“进了球但 reward 是 0”这种错误

如果这四点没过，不要进入 PPO。

### 5.2 单步任务不要把 gamma 当核心问题

S1 是单杆终局任务，属于非常短 horizon 任务。

这意味着：

- `gamma` 影响没有长序列任务那么大
- 真正更重要的是动作映射、reward 正确性、环境可解性
- 不要在 S1 阶段过早沉迷 PPO 超参搜索

### 5.3 动作范围不要过大

S1 很容易出现一个问题：动作空间物理上太大，agent 基本一直在乱打。

解决方式：

- 缩小 `phi` 搜索范围
- 缩小 `V0` 范围
- 先用固定球型
- 先证明局部可学，再扩展范围

### 5.4 sparse reward 是正常的，但别一开始就太难

S1 最常见失败模式不是 PPO 不行，而是任务太难。

例如：

- 初始球型太随机
- 角度范围太大
- 力度范围太大
- 进袋判定过严

解决思路：

- 先固定球型
- 先窄角度窗口
- 先窄力度范围
- 等成功率起来再逐步放宽

本质上就是 curriculum 的简化版。

### 5.5 先做评估集，训练和评估分开

至少维护两套局面：

- `train layouts`
- `eval layouts`

即使 S1 很简单，也不要完全只看训练数据。

最小做法：

- 训练用 `five_point_straight_in` 的固定球型
- 评估额外加入 `eight_ball_reference`

否则你很难判断 agent 是真的会了，还是只记住了某一个位置。

### 5.6 至少跑多个 seed

S1 最后进入 PPO 时，建议至少跑 3 个 seed。

因为：

- 单个 seed 偶然性很大
- 成功一次不代表环境和训练设置稳定
- 你后面 S4 会把“多 seed 方向一致”作为基线要求

## 6. 结果如何观测

S1 不应该只看一条 reward 曲线。至少要同时看下面几类结果。

### 6.1 训练指标

建议记录：

- `episode_reward_mean`
- `success_rate`
- `cue_scratch_rate`
- `illegal_first_hit_rate`
- `phi` 分布
- `V0` 分布
- policy loss
- value loss
- entropy

其中最核心的是：

- `success_rate`
- `episode_reward_mean`

### 6.2 评估指标

固定频率做 eval，例如每 `5k` 或 `10k` environment steps。

每次 eval 记录：

- 固定球型 success rate
- 多个固定球型平均 success rate
- random baseline 对比
- heuristic baseline 对比
- 每个 preset / layout 的 success 明细

### 6.3 可视化观测

S1 非常适合做可视化抽查。

建议每轮 eval 保存：

- 若干成功案例截图 / 动画
- 若干失败案例截图 / 动画
- 命中但未进袋的典型案例
- 母球落袋案例

这样你能快速判断失败是来自：

- 角度偏差
- 力度不对
- first-hit 判定有误
- 物理模拟异常

### 6.4 最重要的三张图

建议最少画这三张：

1. `success_rate vs training_steps`
2. `eval_reward vs training_steps`
3. `phi, V0` 的动作分布图

如果 success rate 不升，动作分布能直接告诉你 policy 是否塌缩到某个坏区域。

## 7. 如何记录实验

S1 阶段就应该建立统一的实验记录习惯，不然后面会乱。

### 7.1 每次实验都要固定保存这些东西

每个 run 目录建议保存：

- `config.yaml`：环境参数 + PPO 参数 + seed
- `train.log`：训练日志
- `eval.json` 或 `eval.csv`：评估结果
- `checkpoints/`：模型权重
- `plots/`：曲线图
- `media/`：成功 / 失败样例动画或截图
- `notes.md`：这次实验的人工结论

### 7.2 训练日志至少记录这些字段

建议每个 run 记录：

- `run_id`
- `date`
- `stage = S1`
- `preset_set`
- `obs_version`
- `action_version`
- `reward_version`
- `seed`
- `total_env_steps`
- `train_success_rate`
- `eval_success_rate`
- `eval_reward`
- `best_checkpoint`
- `remarks`

### 7.3 episode 级别日志建议字段

如果你想做更细分析，可以额外写 episode 级别 CSV/JSONL：

- `episode_id`
- `layout_id`
- `preset_name`
- `phi`
- `V0`
- `success`
- `cue_scratch`
- `legal_first_hit`
- `reward`
- `obj_final_x`
- `obj_final_y`
- `cue_final_x`
- `cue_final_y`
- `termination_reason`

这能帮助你后面做 failure clustering。

### 7.4 checkpoint 保存规则

不要只按时间存 checkpoint。

建议同时支持两种规则：

- `latest checkpoint`
- `best eval success checkpoint`

S1 最有价值的模型，通常不是最后一个，而是 eval success 最好的那个。

按下面顺序做最稳。

### 第一步

先写一个脚本，人工指定球型和 `phi, V0`，验证：

把脚本封装成裸环境：

- `reset`
- `step`
- `reward`
- `done`
- `info`

### 第三步

做 baseline：

- random
- heuristic / grid search

### 第四步

把环境包成 Gymnasium 接口。

### 第五步

### 第六步

固定频率 eval，保存 checkpoint 和案例可视化。

### 第七步

多 seed 复现，确认方向一致。

## 10. 常见坑

### 坑 1：环境没问题吗

很多时候不是 PPO 有问题，而是：

- action 映射错了
- 事件解析错了
- reward 判定错了
- 目标袋口判错了

所以 baseline 非常重要。

### 坑 2：动作空间太大

一上来开放太大的角度和力度范围，会让 agent 长期处于纯随机区。

### 坑 3：只看 reward 不看 success

reward 有时会误导。S1 最核心指标是 `success rate`。

### 坑 4：没有保存失败案例

只看数值曲线很难发现几何问题。一定要保存失败样例可视化。

### 坑 5：训练集和评估集混在一起

这样会让你误以为学会了，实际上只是记住了固定位置。

## 12. 一句话总结

S1 的本质不是“训练一个会打台球的模型”，而是：

**把“固定 preset 球型 + 最小动作空间 + 正确 reward + 稳定评估 + 完整日志”这五件事打通。**

只要这一步做扎实，后面的 S2/S3/S4 就会顺很多。
